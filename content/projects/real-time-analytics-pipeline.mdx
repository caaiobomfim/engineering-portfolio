## Visão Geral

Pipeline de dados construído para ingerir, processar e disponibilizar eventos de comportamento de usuários em tempo real. O sistema processa cerca de 50 mil eventos por segundo em produção.

## Problema

O sistema legado de analytics usava um pipeline batch com execução a cada 6 horas. Isso gerava:

- Dados de dashboard desatualizados para a equipe de produto
- Incapacidade de reagir a anomalias em tempo real (ex: spike de erros)
- ETL frágil que falhava silenciosamente sem alertas

## Stack

| Camada | Tecnologia |
|--------|-----------|
| Ingestão | Apache Kafka |
| Processamento | Python + Faust (stream processing) |
| Armazenamento analítico | ClickHouse |
| Orquestração | Kubernetes (GKE) |
| Monitoramento | Grafana + Prometheus |

## Demonstração

![Arquitetura do Pipeline](/images/placeholder-analytics.svg)

Fluxo: **Producers** (serviços de produto) → **Kafka** (buffer e durabilidade) → **Faust workers** (enriquecimento e normalização) → **ClickHouse** (armazenamento OLAP) → **Dashboards** (Grafana / API interna).

## Resultados

- Latência de ponta a ponta (evento → dashboard): **de 6h para < 1s**
- Throughput sustentado: **50k eventos/segundo** com 8 partições Kafka
- Custo de infraestrutura de analytics: redução de **35%** vs solução anterior (Redshift)
- Uptime nos últimos 6 meses: **99.94%**

## Aprendizados

**ClickHouse como banco analítico:** A escolha do ClickHouse foi decisiva. Queries que levavam 8s no PostgreSQL rodam em 200ms. A curva de aprendizado é alta mas o retorno é imediato para workloads analíticos.

**Kafka para tolerância a falhas:** Antes de Kafka, perder um worker significava perder eventos. Com Kafka, consumidores reiniciam do offset correto. Essa garantia de entrega mudou completamente a confiabilidade do sistema.

**Monitoramento de lag:** A métrica mais importante do pipeline não é throughput — é o consumer lag do Kafka. Essa foi a lição mais cara: descobrimos um gargalo 3 dias depois que já havia causado perda de dados históricos.
