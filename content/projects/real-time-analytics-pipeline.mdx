## Contexto

A equipe de produto dependia de dados de comportamento de usuários para tomar decisões sobre features e experimentos A/B. O pipeline existente era baseado em batch jobs que rodavam a cada 6 horas, o que tornava os dados de dashboard sempre defasados. O volume de eventos em produção era de aproximadamente 50 mil por segundo no pico.

## Problema

O pipeline batch gerava três problemas críticos para o negócio:

- **Latência de dados:** métricas de experimentos A/B chegavam com 6h de atraso, impedindo reações rápidas a comportamentos inesperados
- **Falhas silenciosas:** o ETL legado falhava sem alertas, e os dashboards mostravam dados desatualizados sem indicação de erro para os usuários
- **Incapacidade de reação em tempo real:** anomalias como spike de erros ou queda de conversão só eram detectadas horas depois de ocorrer

## Solução

Substituição do pipeline batch por uma arquitetura de stream processing com Kafka como buffer de eventos e ClickHouse como banco analítico OLAP. O objetivo era reduzir a latência de ponta a ponta de 6 horas para menos de 1 segundo, mantendo a mesma confiabilidade de entrega.

## Arquitetura

```
Serviços de produto (Producers)
    │  eventos JSON via HTTP/Kafka SDK
    ▼
Apache Kafka (8 partições)
    │  buffer durável, retenção 7 dias
    ▼
Faust Workers (Python, 4 réplicas)
    │  enriquecimento, normalização, deduplicação
    ▼
ClickHouse
    │  armazenamento OLAP, particionamento por data
    ▼
Grafana / API interna
    └  dashboards e consultas ad-hoc
```

![Arquitetura do Pipeline](/images/projects/real-time-analytics-pipeline/cover.svg)

O pipeline foi projetado para tolerar falhas: se um worker cai, ele reinicia do último offset commitado no Kafka sem perda de eventos.

## Stack Utilizada

- **Apache Kafka** — buffer de eventos com garantia de entrega e retenção durável
- **Python + Faust** — stream processing; Faust foi escolhido por ser nativo Python e ter boa integração com Kafka
- **ClickHouse** — banco OLAP; queries analíticas que levavam 8s no PostgreSQL rodam em 200ms
- **Kubernetes (GKE)** — orquestração dos workers com autoscaling horizontal
- **Grafana + Prometheus** — monitoramento de saúde do pipeline e métricas de negócio

## Decisões Técnicas

**Kafka vs RabbitMQ:** Kafka foi escolhido pela capacidade de replay de mensagens (consumidores podem reprocessar eventos históricos) e pela separação entre produção e consumo — múltiplos consumers independentes podem ler o mesmo tópico sem interferência.

**ClickHouse vs PostgreSQL para analytics:** PostgreSQL não é adequado para queries analíticas de alta cardinalidade em datasets grandes. ClickHouse usa armazenamento colunar e compressão agressiva, o que reduziu o tamanho do storage em ~70% e acelerou queries em 40x.

**Deduplicação no worker, não no banco:** Eventos duplicados são filtrados no Faust worker usando uma janela de tempo + hash do evento, antes de chegar ao ClickHouse. Deduplicar no banco seria mais caro e mais lento.

## Desafios

**Consumer lag como indicador de saúde:** A métrica mais importante do pipeline não é throughput — é o consumer lag do Kafka. Descobrimos isso da pior forma: um worker com bug estava processando eventos mas não commitando offsets, causando reprocessamento em loop. O lag explodia enquanto o throughput parecia normal.

**Particionamento do ClickHouse:** A escolha errada de chave de particionamento gerou hotspots em produção. Particionar por `event_type` concentrou escrita em poucas partições durante picos. A correção para particionamento por data resolveu o problema mas exigiu migração dos dados existentes.

## Aprendizados

**Observabilidade de pipeline é diferente de observabilidade de API.** Métricas de latência e error rate não são suficientes — é preciso monitorar lag de consumidor, tamanho de batch, taxa de deduplicação e tamanho de partições para ter visibilidade real do que está acontecendo.

**Stream processing tem um modelo mental diferente de request/response.** A curva de aprendizado da equipe foi maior do que o esperado. Documentar o modelo de garantias (at-least-once, exactly-once) e os contratos de cada tópico Kafka reduziu significativamente os bugs de integração.
