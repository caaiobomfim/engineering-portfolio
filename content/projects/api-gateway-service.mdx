## Contexto

A plataforma era composta por dezenas de microsserviços desenvolvidos de forma independente ao longo de 2 anos. Cada equipe expunha seus serviços diretamente, sem um ponto de entrada centralizado. O ambiente era produção com tráfego real e times distribuídos.

## Problema

A ausência de um gateway causava problemas concretos em três frentes:

- **Autenticação duplicada:** cada microsserviço implementava sua própria verificação de token JWT, com lógicas ligeiramente diferentes, gerando inconsistências e bugs de segurança
- **Rate limiting inexistente:** qualquer cliente podia fazer chamadas ilimitadas a qualquer serviço, expondo a plataforma a abusos e consumo excessivo de recursos
- **Observabilidade zero:** sem ponto central, era impossível rastrear o ciclo completo de uma requisição entre serviços

## Solução

Construção de um API Gateway em Go que atua como ponto único de entrada. Toda requisição externa passa pelo gateway antes de chegar a qualquer serviço interno. O gateway é responsável por autenticação, roteamento e rate limiting — os serviços internos passam a assumir que qualquer requisição recebida já foi autenticada.

## Arquitetura

```
Cliente externo
    │
    ▼
API Gateway (Go)
    ├── Middleware: validação JWT via Redis
    ├── Middleware: rate limiting (sliding window)
    ├── Middleware: injeção de trace ID
    │
    ▼
Roteamento por rota/header
    ├── → Service A (gRPC)
    ├── → Service B (gRPC)
    └── → Service C (gRPC)
```

O gateway não faz lógica de negócio — apenas valida, enriquece e roteia. Os trace IDs são injetados no header de cada requisição para rastreamento end-to-end.

## Stack Utilizada

- **Go 1.22** — linguagem principal; escolhida por performance e simplicidade no modelo de concorrência
- **gRPC** — comunicação interna entre o gateway e os microsserviços
- **Redis** — cache de tokens validados e implementação de rate limiting
- **Docker + Compose** — containerização e orquestração local
- **Envoy** — proxy reverso na borda antes do gateway

## Decisões Técnicas

**gRPC internamente vs REST:** A comunicação entre o gateway e os serviços foi migrada de REST para gRPC. O motivo principal foi a tipagem forte via Protobuf — contratos bem definidos eliminaram uma classe inteira de bugs causados por divergências entre o que o serviço esperava e o que o cliente enviava.

**Redis para cache de tokens:** Em vez de revalidar o JWT criptograficamente a cada requisição, o resultado da validação é cacheado no Redis com TTL igual ao tempo de expiração do token. Isso reduziu a latência de autenticação de ~8ms para ~1ms em p99.

**Rate limiting com sliding window:** Algoritmo escolhido por ser mais justo que fixed window (que permite bursts no limite de janelas adjacentes). Implementado com comandos atômicos no Redis para evitar race conditions em múltiplas instâncias do gateway.

## Desafios

**Migração zero-downtime:** Os serviços existentes não podiam ter downtime durante a adoção do gateway. A solução foi um período de shadow mode onde o gateway roteava as requisições mas os serviços ainda faziam sua própria autenticação, até que a confiança no gateway fosse estabelecida.

**Propagação de trace ID:** Garantir que todos os serviços internos propagassem o trace ID nas chamadas subsequentes exigiu padronização de headers e alterações em todos os clientes gRPC existentes.

## Aprendizados

**Observabilidade deve ser o primeiro requisito, não o último.** A decisão de injetar trace IDs desde o primeiro deploy foi a que gerou mais valor no longo prazo. Sem ela, cada incidente era investigado de forma cega.

**Centralizar autenticação em um único lugar tem um custo:** o gateway vira ponto único de falha. O investimento em alta disponibilidade e circuit breaker no gateway foi necessário e não estava no escopo original.
